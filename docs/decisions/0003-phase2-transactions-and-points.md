# ADR-0003: フェーズ2の取引状態とおかげさまポイント台帳

- 状態: 一部置換済み（ADR-0005）
- 日付: 2026-07-20
- 対象: 取引状態、完了確認、ポイント台帳、共通おかげさまプール

## 背景

> 2026-07-21追記: 所有権移転とプラットフォーム上の完了の区別、ポイント上限、有効期限、保有上限、共通プール移行、遡及適用はADR-0005を正とする。本ADRの自動移行未実装という実装状況はそのままだが、仕様上の未決事項ではなくなった。

既存文書では取引状態とポイントの原則は定義されているが、C-106（受取人選択後の承諾）、保留後の解決手順、共通プールへの具体的な移行時期は人間の判断待ちである。本ADRは法的判断や本番運用の確定ではなく、フェーズ2を安全側で検証するための暫定実装を定める。

## 決定

### 取引生成と状態遷移

- 提供者が受取申込みを選択した同一DBトランザクション内で、取引を `RECIPIENT_SELECTED` として1件生成する。
- 受取人本人の承諾で `ACCEPTED`、当事者の受渡し準備確認で `HANDOVER_SCHEDULED` とする。実際の住所や電話番号、価格、送料額は保存しない。
- 提供者・受取人の完了報告時刻を別々に保持する。片方の報告状態を経て、双方が報告した時だけ `UNDER_ADMIN_REVIEW` に進む。
- 管理確認前は物品を完了状態にせず、ポイント残高へ影響する仕訳を作らない。
- 状態変更は用途別サービスコマンドだけで行い、`status` の汎用更新APIは作らない。
- 状態イベントは追記型とし、更新・削除をDBトリガーで拒否する。

### ポイント付与

- 管理者が双方の完了報告を確認し、取引を `COMPLETED` にする同一DBトランザクションで提供者へ基本1ポイントを記録する。
- 配送協力加算は金額ではなく作業区分 `NONE=0 / SIMPLE=1 / STANDARD=2 / LARGE_SPECIAL=3` から決定する。
- 対面手渡しの取引では加算を0に制限する。基本を含む合計は最大4ポイントとする。
- 基本付与と配送加算には取引単位の一意な冪等キーを付け、二重付与をDB制約でも拒否する。

### 追記型台帳と保留・取消し

- 利用者ポイント残高カラムを作らない。表示残高は `POSTED` 仕訳の符号付き合計から導出する。
- 管理確認で保留した場合は `HELD` 仕訳を追記する。`HELD` は表示残高へ算入しない。
- 保留解消後に承認する場合は新しい `POSTED` 仕訳を追記し、既存の `HELD` 記録は変更しない。
- 確定済み付与の取消しは、元仕訳と同じ絶対値・逆符号の `REVERSAL` 仕訳を追記する。元仕訳は変更しない。
- `reversal_of` を一意にし、同じ元仕訳への二重取消しを拒否する。
- 管理者が残高や既存台帳行を直接変更する画面・サービス・APIは作らない。

### 共通おかげさまプール

- 現金・寄付原資とは別の追記型 `common_pool_ledger_entries` として管理し、残高カラムを作らない。
- 期限切れ、未指定、保有上限超過の具体条件は未決のため、自動移行は実装しない。
- フェーズ2では管理者が理由区分と説明を指定し、対象の確定ポイント仕訳を全額だけ共通プールへ移すコマンドを用意する。利用者自身が他者やプールへ送る機能は作らない。
- 移行時は利用者台帳へ負数仕訳、プール台帳へ同額の正数仕訳を同一DBトランザクションで追記する。

### 権限

- 当事者コマンドは取引の提供者または受取人本人だけが実行できる。
- 完了確認・付与保留は `MODERATOR` または `ADMINISTRATOR` に許可する。
- 確定ポイントの取消しと共通プール移行は、暫定的に `ADMINISTRATOR` だけに限定する。
- `AUDITOR` は取引・台帳を閲覧できるが変更できない。

## 実装しないもの

- ポイントによる物品・サービス取得
- ポイントの購入、換金、売買、会員間譲渡・送金
- 円換算率、物品価格、送料額
- ポイントと寄付用現金の同一残高管理
- 共通プールへの自動移行スケジュール
- 配送会社APIおよび送料精算

## 見直し条件

C-106、ポイント期限・保有上限、共通プール移行条件、保留解消SLA、管理権限マトリクスが承認された時点で見直す。
