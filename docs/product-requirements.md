# プロダクト要件定義（MVP）

- プロダクト名: 物品循環プラットフォーム／藍の夕凪（仮称）
- 対象地域: 岡山県倉敷市および周辺地域（実証運用）
- 文書状態: 制度資料・正式決定反映版。未決事項は `requirements-conflicts.md` に分離。
- 要件源: 2026-07-20の依頼、`docs` 配下の制度資料、ADR-0004、ADR-0005
- 最終更新: 2026-07-21

## 1. 目的と成功条件

不要になった物品を必要な人へ無償で直接譲渡できる、安全性と透明性を重視した仲介・取引管理基盤を提供する。通常の無償譲渡の完了を社会貢献の応援票につなげるが、物品、ポイント、現金寄付を明確に分離する。

MVPの成功指標は未決である。少なくとも「成立取引数」だけで評価せず、安全性（通報率、禁止品検知、紛争、管理確認時間）、完了率、寄付公開の正確性を含むKPIを運用開始前に定義する。

初期MVPは全国公開を目的とせず、岡山県倉敷市および周辺地域で、招待制・登録上限50名の小規模実証として運用する。全国公開機能はfeature flagを既定で無効にする。「周辺地域」の具体範囲と登録上限の算入対象は運用開始前の承認事項とし、推測で本番設定へ入れない。

## 2. 絶対に守る境界

### 2.1 運営者が原則行わないこと

- 物品の買い取り、所有、転売、価格決定、通常時の譲渡先決定
- 物品代金の受領
- 利用者間の金銭決済

### 2.2 利用者間で禁止すること

- 現金、送料、金券、外部ポイント、他の物品、役務、謝礼など、名目を問わない対価の授受・交渉
- 金額、希望価格、希望ポイントを掲載条件または受取人選定条件にすること

### 2.3 おかげさまポイントで禁止すること

- 換金、金券・外部ポイント交換、購入、売買、会員間移転、他人への送信
- 物品・サービス取得への利用
- 固定円換算率、物品価格・市場価格・送料額との連動
- 受取人の選定条件への利用

画面文言にも「○円相当」「還元」「送料補填」「報酬」等の誤解を招く表現を使わない。法的評価を断定する表示も行わない。

### 2.4 所有権とプラットフォーム上の取引確定

- 所有権移転は当事者間の合意および現実の引渡しによる。
- 双方の完了報告は、プラットフォーム上の取引完了報告として記録する。
- 運営者確認は、運営上の取引確定とポイント付与の条件である。
- 運営者確認を所有権移転の要件として表示・説明しない。
- 当事者報告、双方報告済み、運営確認済みを別の時刻・イベント・表示として扱う。

## 3. 利用者とロール

| ロール            | 主な責務                                               | 制約                                    |
| ----------------- | ------------------------------------------------------ | --------------------------------------- |
| user              | 掲載、申込み、取引、メッセージ、完了報告、ポイント配分 | 自分と参加取引に限定                    |
| moderator         | 投稿・取引・メッセージ・通報の審査                     | 寄付原資承認は不可を暫定提案            |
| donation_reviewer | 団体審査、寄付期間・計算・送金記録の確認               | 会員制裁は不可を暫定提案                |
| administrator     | 設定・権限・重要操作                                   | 監査対象。ポイント残高直接変更不可      |
| auditor           | 監査ログ・台帳・公開根拠の閲覧                         | 原則読取専用。必要最小限のPIIマスキング |

詳細権限マトリクスは未決（C-115）。「管理者」という画面上の総称と、各ロールの実権限を分ける。

## 4. MVPスコープ

### 4.1 アカウント

- 会員登録、ログイン、ログアウト、メール確認、パスワード再設定
- 運営者発行の単回利用招待コードによる招待制登録。利用者による自由な再招待は提供しない
- 登録上限はサーバー側設定値で管理し、初期値50名。全国公開feature flagは既定で無効
- 18歳以上の個人だけを対象とし、1人1アカウント。法人・団体アカウントは初期MVP対象外
- 招待元、発行者、発行・使用・失効日時、使用者を監査可能に記録。生コードをログ・監査へ保存しない
- 利用規約同意、プライバシーポリシー同意、18歳以上の確認を文書版・時刻・確認経路とともに追記記録
- プロフィールの閲覧・更新
- 住所・本人確認情報を通常プロフィールから分離
- KYC状態 `unverified` / `pending` / `verified` / `rejected` をアダプター経由で管理
- 開発環境は管理者が状態を変更できるモック。実画像をseedしない
- 閲覧と仮登録はKYC不要。出品、受取申込み、受取人承諾、日程合意、取引メッセージ、完了報告には有効な `verified` KYCを要求
- 本番KYC導入前はfeature flagとモックを使用する。本番でモックまたは設定欠落の場合はKYC必須操作をfail closed
- 退会、個人情報開示・訂正・削除/利用停止相当の申請受付と処理履歴

### 4.2 物品掲載と検索

- 必須項目: `title`, `description`, `category`, `condition`, `defect_description`, `images`, `delivery_method`, `handover_area`, `available_dates`, `shipping_supported`, `status`, `owner_user_id`, timestamps
- 状態: `draft`, `pending_review`, `published`, `reserved`, `handover_in_progress`, `completed`, `rejected`, `cancelled`, `suspended`
- 価格・希望ポイント等の入力欄を持たない
- 許可カテゴリ/禁止ルールによる事前検証と管理者審査
- 制度資料の禁止15分類を版管理されたルールとして実装し、初期は高リスク品を広く除外する。初期許可カテゴリーと例外審査権限は公開前に承認する
- 物品の新規登録、下書き保存、投稿申請には有効な `verified` KYCをサーバー側で要求する。既存下書きの参照可否はKYC失効時ポリシーに従う

### 4.3 申込みと取引

- 公開物品への「ほしい」申込み
- 申込み、受取人承諾、取引参加には有効な `verified` KYCを要求する
- 1掲載への複数申込みと、提供者による1受取人選択
- 取引当事者だけのメッセージ
- 対面手渡しを基本とする日程調整
- 双方の完了報告、管理者確認、紛争・取消し・停止
- 正確な受渡情報は、受取人が承諾して取引が成立した後、その取引当事者だけに必要最小限開示する
- 双方の報告を当事者によるプラットフォーム上の完了報告として記録し、管理者確認を運営確認済み状態とポイント付与条件として扱う
- 状態遷移はサービス層がサーバー側で検証する
- 同一アカウントは自己申込み不可。同一人物・不自然な関係の判断材料を管理画面へ提示するが、自動断定しない

### 4.4 メッセージ・安全機能

- 金銭交渉語句を設定データで検知し、ルールごとに警告または保留
- 初期語句: 現金、振込、PayPay、商品券、ギフト券、送料別、謝礼、円、値段、買う、売る
- 検知イベント、該当ルール版、判断、判断者、理由を保存
- 自動検知だけで永久停止しない
- 通報、ブロック、警告、一時停止、永久停止、異議申立て

### 4.5 配送

- 提供者が任意選択した場合のみ配送情報を登録
- 利用者間の送料を含む決済・請求・精算機能を持たない
- `carrier`, `tracking_number`, `shipping_method`, `shipped_at`, `delivery_status`, `delivered_at`, `recipient_confirmed_at`, `shipping_workload_level`, `admin_verified_at` を保持
- 追跡番号は暗号化し、表示・監査・ログを制限
- 配送APIはインターフェースと開発モックのみ

### 4.6 ポイント

- 通常取引完了を管理者が確認した後、提供者へ基本1ポイント
- 配送協力加算は作業区分 `none=0` / `simple=1` / `standard=2` / `large_special=3` から決め、上限3ポイント
- 基本1ポイントを含む1取引合計は1〜4ポイント。対面手渡しのみで配送作業がない場合の加算は0
- 追記型台帳で付与、保留、確定、取消し、反対仕訳を表現
- 最低項目: `transaction_id`, `user_id`, `event_type`, `points`, `reason`, `created_at`, `created_by`, `reversal_of`, `status`
- 残高は確定済み台帳の集計値。キャッシュを持つ場合も台帳と再構築可能にする
- 架空、取消し、未着、禁止品、金銭授受の疑いは保留/取消しの対象だが、自動断定しない
- 正式ポイントは付与日の1年後が属する月の末日に失効。Asia/Tokyoで翌月1日00:00から利用不可
- 失効予定を60日前、30日前、7日前に通知できる構造を持つ
- 利用者の利用可能残高上限は30ポイント。確定付与で超える部分は共通おかげさまプールへ同一DBトランザクションで移す
- 失効・上限超過移行は利用者台帳の負数仕訳とプール台帳の正数仕訳を追記し、元仕訳を変更しない
- 本番開始前の開発データは正式ポイントに含めない。本ルールは本番開始日時以後の新規正式付与に適用し、将来変更も別決定なしに遡及しない

### 4.7 寄付配分

- 審査済み支援団体の閲覧
- 利用者がポイントで応援先を指定
- 団体別のポイント比率集計
- 管理者がポイントとは別に寄付原資（整数円）を登録
- 予定額を決定的ルールで計算し、承認、送金記録、証明資料登録、公開
- 状態: `draft`, `allocation_open`, `allocation_closed`, `calculated`, `approved`, `remittance_in_progress`, `completed`, `suspended`, `cancelled`
- 1ポイント当たり金額を保存・表示しない
- 未指定、期限切れ、上限超過を共通プールへ追記型で移せる構造。期限切れと上限超過はADR-0005の条件で処理し、未指定の具体条件は別途決定
- ポイント台帳と現金予算・送金台帳を別テーブル・別ドメインにする

### 4.8 管理・監査

- 会員/KYC、投稿、禁止品、取引、ポイント台帳、検知、通報、制裁、異議の管理
- 団体の登録・審査・停止、寄付期間、原資、計算、承認、送金、証明、公開
- 監査ログと権限別CSVエクスポート
- 重要操作は実行者、日時、対象、操作、変更前後（PIIを除く/マスク）、理由、相関IDを記録
- 管理者によるポイント残高直接上書きを設計・UI・APIすべてで禁止

### 4.9 公開透明性ページ

- 寄付対象期間、確定原資額、原資区分
- 団体別配分率・寄付額、送金日・状態、公開可能な証明資料
- 繰越額、不正取引による取消件数、更新日時
- 配分だけの段階を「寄付完了」と表示しない
- 公開スナップショットを版管理し、後日の再計算で過去表示を無断上書きしない

## 5. 主要業務ルール

| ID     | ルール                                     | 検証場所                        |
| ------ | ------------------------------------------ | ------------------------------- |
| BR-001 | 掲載所有者は自分の物品へ申込みできない     | DB制約可能範囲 + 取引サービス   |
| BR-002 | 公開・受付可能な掲載だけ申込み可能         | 取引サービス + トランザクション |
| BR-003 | 1掲載につき有効な選択受取人/取引は最大1    | DB部分一意制約 + 取引サービス   |
| BR-004 | 任意の状態値をクライアントから直接更新不可 | 専用コマンドAPI + 状態機械      |
| BR-005 | ポイント確定は取引完了・管理確認後のみ     | ポイントサービス                |
| BR-006 | 取引由来付与は基本1 + 加算0..3、合計1..4   | ポイントサービス + DBチェック   |
| BR-007 | 台帳記録を更新/削除して訂正しない          | DB権限 + サービス +監査         |
| BR-008 | ポイント配分で商品/サービスを得られない    | UI非提供 + API不在 + 規約       |
| BR-009 | 寄付原資・送金とポイントを結合残高にしない | モジュール/テーブル分離         |
| BR-010 | モデレーション検知だけで永久停止しない     | 制裁サービスのガード            |
| BR-011 | auditorは原則変更操作不可                  | サーバー認可 + DB権限の検討     |
| BR-012 | 重要管理操作に理由を必須とする             | 入力スキーマ + 監査サービス     |
| BR-013 | 出品・申込み・取引参加はverified KYC必須   | 認可ポリシー + サービスguard    |
| BR-014 | 招待コードは運営発行・単回利用             | 招待サービス + DB制約 + 監査    |
| BR-015 | 実証登録上限は設定値、初期50名             | 登録サービス + 排他制御         |
| BR-016 | 利用者は18歳以上の個人、1人1アカウント     | 登録/KYCポリシー + 審査         |
| BR-017 | 正確な受渡情報は成立後の当事者だけへ開示   | サーバー認可 + PII分離          |
| BR-018 | 利用可能ポイントは最大30                   | 台帳サービス + 原子的pool移行   |
| BR-019 | ポイントは1年後の月末に追記失効            | 期限ジョブ + 台帳 + outbox      |
| BR-020 | 運営確認を所有権移転の条件と表示しない     | 用語辞書 + API/UIテスト         |

## 6. 非機能要件

- 日本語優先、レスポンシブ、本文16px相当以上を基準候補、十分なコントラスト
- WCAG 2.2 AAを目標（適合宣言の範囲と試験は未決）
- キーボード操作、明確なフォーカス、label、色以外の状態表現、具体的な日本語エラー
- UTC保存・日本時間表示。締切はタイムゾーン付きで保持
- 金額は浮動小数でなく整数円、ポイントは整数
- PIIをアプリログ・分析イベントに出さない
- 重要処理は冪等キー、DBトランザクション、楽観/悲観ロックを用途別に使用
- 本番秘密は環境変数/シークレット管理。開発seedは架空データのみ
- 可用性、性能、保持期間、バックアップ目標はC-004として未決

## 7. MVPの対象外

- 物品代金・送料・手数料など利用者間の決済
- ポイントの購入、換金、交換、送信、譲渡、商品取得
- 配送会社・KYC事業者との本番接続
- 全国公開、一般開放登録、利用者からの再招待
- 法人・団体アカウント
- 運営による物品保管、倉庫、買い取り、再販売
- 自動AIだけによる永久停止・法的判断
- ネイティブモバイルアプリ、多言語、推薦最適化（将来候補）

## 8. リリースゲート

本番公開には最低限、以下を要求する。

1. 受領した制度資料の版管理、相談版と公開版の分離、残存placeholderの解消
2. L-001〜L-012のうち適用する法務・税務確認の記録
3. 禁止品、保持期間、権限マトリクス、周辺地域の範囲、登録上限算入対象、1人1アカウント照合方法の承認
4. 脅威モデリング、依存関係/秘密/アップロード検査、復旧試験
5. 主要フローE2E、状態遷移、台帳不変条件、寄付計算のテスト合格
6. 管理運用手順、インシデント対応、問い合わせ・異議申立て体制
7. 招待・50名上限・全国公開off・KYC fail closedの設定試験
8. 本番開始日時とポイントポリシー版の承認、開発ポイント除外の照合
