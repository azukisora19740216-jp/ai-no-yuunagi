# セキュリティ・プライバシー方針

- 状態: ADR-0004・ADR-0005反映版の設計方針。法務見解・認証取得・安全性保証ではない。
- 基準: 最小権限、デフォルト拒否、データ最小化、追跡可能性、秘密と公開データの分離。

### フェーズ2実装注記

- 取引参加者IDとサーバー側ロールの双方でコマンド権限を検証する。
- 管理確認・保留は moderator / administrator、反対仕訳・共通プール移行は administrator に限定する。
- 取引状態はバージョン付き条件更新で競合を検出する。
- 二重付与は状態guard、冪等キー、一意制約の三層で拒否する。
- 追記型台帳と状態イベントはアプリ権限に加えDBトリガーでも更新・削除を拒否する。
- 監査差分にはポイント数、状態、作業区分だけを保存し、メッセージ本文や住所を含めない。

## 1. 保護対象と主な脅威

| 保護対象               | 主な脅威                                  | 重点対策                                                |
| ---------------------- | ----------------------------------------- | ------------------------------------------------------- |
| アカウント・セッション | credential stuffing、乗っ取り、管理者悪用 | 強いハッシュ、レート制限、セッション失効、管理者MFA候補 |
| 住所・KYC・追跡番号    | 漏えい、内部者閲覧、ログ流出              | 分離、暗号化、最小表示、アクセス監査、保持制御          |
| 物品画像・証明資料     | マルウェア、位置情報、口座情報公開        | 隔離、再エンコード、EXIF除去、原本/公開版分離           |
| メッセージ             | 金銭交渉、嫌がらせ、XSS、過剰監視         | 出力エスケープ、設定型検知、人手審査、閲覧監査          |
| ポイント台帳           | 二重付与、直接改ざん、競合、管理者不正    | 追記専用、冪等性、DB制約、反対仕訳、照合                |
| 寄付原資・送金         | ポイントとの混同、端数改ざん、証憑差替え  | ドメイン分離、二者承認、計算snapshot、checksum          |
| CSV                    | 大量PII持出し                             | 列最小化、権限、非同期、短期URL、暗号化、監査           |

## 2. 必須コントロール

### 認証・パスワード

- パスワードを保持する場合はArgon2idを第一候補とし、OWASP等の現行指針と本番リソースでパラメータを決める。平文・復号可能形式・ログ出力は禁止。
- メール確認/再設定トークンは十分な乱数、短期失効、単回利用、DBにはハッシュを保存。
- ログイン、登録、再設定、確認再送はIP・アカウント・端末軸の段階的レート制限。具体値は負荷/運用要件後に設定。
- セッションIDをログに出さない。権限変更、停止、パスワード変更、退会時に関連セッションを失効。
- 管理者はMFAと短いセッションを本番公開ゲート候補とする。
- 招待コードは十分な乱数、単回利用、短期失効、DBにはハッシュだけを保存する。生コードをURLログ、監査、分析へ出さない。
- 招待使用と登録上限50の枠確保を原子的に行い、同時登録で51人目を作らない。障害時にopen registrationへfallbackしない。
- 規約、プライバシーポリシー、18歳以上確認は版・時刻・経路を追記記録し、既存証跡を上書きしない。

### CSRF・XSS・SQLインジェクション

- Cookieセッションの変更系リクエストはCSRFトークンまたはフレームワークの検証済み機構に加え、Origin/Hostを検査。
- Reactの既定エスケープを維持し、`dangerouslySetInnerHTML` を原則禁止。CSPはNext.js Proxyでリクエストごとの予測困難なnonceを生成し、framework scriptにも同じnonceを付与する。productionの `script-src` に `unsafe-inline` / `unsafe-eval` を許可しない。
- URL、Markdown、画像参照を許可リストでサニタイズ。アップロードSVG/HTMLを拒否。
- Prismaのparameterized APIを使い、生SQLはプレースホルダ必須。識別子の文字列連結を禁止。

### 入力・ファイル

- すべての外部入力をサーバー側で型、長さ、文字種、列挙値、相互条件まで検証。
- ファイル名を保存キーに使わず、ランダムID化。サイズ/個数/画像寸法/実MIME/マジックバイトを検査。
- 画像をデコードして安全形式へ再エンコードし、EXIFを除去。検査前は隔離し、同一originで直接配信しない。
- 証明資料は公開前に墨消し版を作り、別担当承認。公開後のURL推測を防ぐ。

### 認可

- UI非表示は認可ではない。各サービスコマンドでロール、権限、対象所有権、取引参加、アカウント状態を検証。
- 閲覧・仮登録はKYC不要。物品登録・下書き保存・投稿申請、受取申込み、受取人承諾、日程合意、取引メッセージ、完了報告は有効なverified KYCを必須とする。
- 全国公開feature flagはサーバー側query/commandでも既定拒否とし、UI非表示だけに依存しない。
- 正確な受渡情報は取引成立後のprovider/recipientだけに開示し、ID差替え、一覧、検索、exportで取得できないようにする。
- auditorは変更系サービスを呼べない。DB直アクセスを許す場合も読取専用・マスクviewを使用。
- 重要操作（権限、制裁、台帳、寄付、証明公開、CSV）は理由必須・監査必須。
- ポイント残高の直接更新API、汎用状態PATCH、管理者向け任意SQL機能を作らない。

## 3. データ分類

| 区分              | 例                                               | 保存・表示方針                        |
| ----------------- | ------------------------------------------------ | ------------------------------------- |
| Public            | 公開掲載、承認済み団体、透明性snapshot           | 公開承認済み版のみ                    |
| Internal          | ルール設定、集計、非PII監査                      | staff権限、監査                       |
| Confidential      | メール、メッセージ、通報、IPリスク情報           | 目的別制限、暗号化、マスク            |
| Highly Restricted | 住所、本人確認参照、追跡番号、団体口座、証明原本 | 分離保存、最小担当者、復号監査        |
| Secret            | パスワードhash、API key、署名鍵、セッション      | secret manager/専用列、絶対にログ禁止 |

公開名や粗い地域も組み合わせにより個人を識別し得る。分類は単一フィールドでなく利用文脈で評価する。

## 4. ログと監査

### アプリケーションログ

- request/correlation ID、ルート識別子、状態コード、処理時間、安全なエラーコードを記録。
- 氏名、住所、メール、電話、メッセージ本文、追跡番号、ファイル内容、token、cookie、認証headerを記録しない。
- 例外を構造化し、本番レスポンスへstack/SQL/環境変数を出さない。

### 監査ログ

- actor、時刻、action、target、理由、安全な変更前後、結果、request IDを追記。
- 監査イベントの更新・削除をアプリ権限から禁止し、アクセスも監査する。
- PII値そのものではなくフィールド名と「変更あり」を保存する。
- 保持期間、WORM/外部保管、ハッシュチェーンは法務・脅威・費用評価後に決定。

## 5. プライバシー運用

- 利用目的と取得項目のデータマップを作り、目的に不要な情報は取得しない。
- 住所とKYCは通常プロフィール、検索インデックス、分析基盤から分離。
- 開示・訂正・削除・利用停止等の申請は本人確認、受付、期限、判断、実行、通知を状態管理。
- 削除申請があっても、紛争、監査、会計、法令等により保持が必要な可能性を区分ごとに審査し、保留根拠と終了日を記録。
- 法令上の保存期間を本書で断定しない。個人情報保護委員会は一律の保存期間を定めていない旨を案内しているため、利用目的・他法令・運用必要性を踏まえ保持表を人間が承認する。
- 本番データを開発・テストへ複製しない。seedは架空の `.invalid` メール、架空地域、ダミートークンだけを使う。
- 本番開始前のpoint ledgerを正式残高へ混入させない。ポリシー版なしの開発行を更新して正式化せず、正式projectionから除外する。

参考: [個人情報保護委員会の通則編ガイドライン](https://www.ppc.go.jp/personalinfo/legal/guidelines_tsusoku/)、[保存期間に関するFAQ](https://www.ppc.go.jp/all_faq_index/faq1-q5-2/)。

## 6. 暗号化と秘密管理

- TLSを必須化し、HSTSは本番ドメイン確認後に段階導入。
- DB/バックアップ/オブジェクト保存のat-rest暗号化に加え、高機密フィールドはアプリ層のenvelope encryptionを検討。
- 鍵にversionを持たせ、ローテーション・再暗号化・失効・緊急交換のrunbookを作る。
- `.env.example` はキー名と安全なダミー値だけ。本番secret、秘密鍵、API keyをGitへ置かない。
- CIはsecret scanningと依存関係脆弱性検査を行う。検出時の自動破棄/ローテーション手順を用意。

## 7. 不正・安全設計

- 同一アカウントの自己取引はDB/サービスで禁止。
- 同一人物/関係性は、KYC reference一致、端末・住所等の必要最小限シグナルとして管理者へ提示するが、自動で断定・永久停止しない。
- 1人1アカウントを目的としても、生年月日・本人確認画像・端末fingerprintを必要性評価なしに追加収集しない。照合キー、例外、保持期間を承認する。
- シグナル生成の利用目的、保持期間、説明可能性、誤検知への異議を定めるまで高侵襲なfingerprintingを導入しない。
- メッセージ語句は版管理設定とし、正規化・部分一致による誤検知をテストする。検知断片はマスク表示。
- 通報者と被通報者の情報を相互に不要開示しない。緊急時対応手順と公的機関対応窓口は運用設計で決める。

## 8. 可用性・バックアップ・インシデント

- DBは暗号化バックアップ、復元テスト、point ledgerと寄付snapshotの整合チェックを行う。
- RPO/RTO、バックアップ頻度/保持、リージョンは未決。C-004後に確定。
- インシデント時は検知、封じ込め、証拠保全、影響評価、通知判断、復旧、再発防止を記録。
- 漏えい等の報告・本人通知要否と期限は個人情報保護委員会の最新ガイドラインをインシデント時および公開前に確認する。
- 依存障害時もポイント二重付与や寄付再送金を避けるため、冪等性と手動照合を優先する。

## 9. テストすべきセキュリティ不変条件

- 他人の住所・KYC・追跡番号・取引メッセージへID変更でアクセスできない。
- auditorおよび権限不足staffが変更APIを実行できない。
- CSRF token/Origin不正の変更要求が拒否される。
- 保存/反射/DOM XSS payloadが実行されない。
- executable/polyglot/巨大画像/偽MIMEが拒否される。
- 同じ冪等キーでポイント・送金・状態遷移が二重実行されない。
- 同じ招待コードを同時使用できず、登録上限50を競合で超えない。
- KYC未確認者がURL/APIを直接操作しても、出品・申込み・取引参加できない。
- 受渡情報を取引成立前、第三者、auditorの通常閲覧から取得できない。
- 全国公開offのとき、公開query/APIを直接呼んでも対象外公開されない。
- 同時ポイント付与でも正式利用可能残高が30を超えず、超過user outとpool inが同額・原子的である。
- 失効処理と60/30/7日前通知が冪等で、元付与仕訳を更新しない。
- 管理者でも残高を直接上書きできない。
- 自動モデレーションだけで永久停止にならない。
- 本番設定でmock KYC/shipping、既定secret、debug errorが起動を失敗させる。
